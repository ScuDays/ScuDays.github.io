    Lecture 23：死锁，有界缓冲区，C++互斥模式 Deadlock, Bounded Buffer，C++ Mutex Patterns¶

    我们研究如何正确同步有界缓冲区数据结构与并发的读写线程。保证一个共享数据在获得互斥锁之后只被一个线程访问。

    为了性能，我们更喜欢将临界区尽可能地缩小：临界区中的代码越多，我们可以获得的并行性就越少，因为只有一个线程可以在临界区中运行。但我们必须确保临界区足够大以使程序正确：它需要包含我们尝试同步的操作中的所有对共享状态的访问（在有界缓冲区的情况下为 read() 和 write() ）。

    死锁可能发生在两个线程试图互相获取已经被锁定的资源的情况下。也可能发生在其他情况下，比如一个线程试图获取它已经持有的锁。

    Deadlock 死锁¶

    锁很难，不仅因为我们需要确保在临界区内包含对所有共享数据的访问，还有可能因为出错，导致我们的程序无限期地卡住——死锁。

    来看例子：

    :::tips 如果程序包含两个线程 T1 和 T2，它们都以不同的顺序锁定互斥锁，则可能会出现如下情况：

    1. T1 尝试锁定 M1，并成功。
    2. T2 同时尝试锁定 M2，并成功。
    3. T1 现在想要锁定 M2，但不能，因为 T2 锁定了 M2。因此，T1 会自行挂起（“块”），直到 M2 可用。
    4. 同时，T2 想要锁定 M1，但不能，因为 T1 锁定了 M1。因此，T2 也会自行挂起，直到 M1 可用。
    5. 两个线程都无法继续，因此两个线程都无法解锁已锁定的互斥锁。

    因此，任何线程都无法再次取得进展，程序将陷入死锁。

    :::


    示例程序：有界缓冲区¶

    :::tips 有界缓冲区是计算机系统中常见的抽象概念：内存是有限的，因此我们需要限制我们用于缓冲区的空间量。然而，我们可能事先不知道需要在缓冲区中存储多少数据，因此我们可能需要在插入更多数据之前允许一些数据从缓冲区中被消耗掉。

    :::

    这个想法可以通过使用循环缓冲区来高效实现，其中写入（和读取）会在末尾绕回到开头。循环缓冲区非常高效，因为它只有在读取或写入字节时才需要在缓冲区内部移动数据（字节）。

    在接下来的示例中，我们使用两个线程：一个写线程和一个读线程。写线程（也称为”生产者”）将数据写入有界缓冲区，而读线程（也称为”消费者”）从中读取数据。

    • 缓冲区可以容纳最多 CAP 个字符，其中 CAP 是缓冲区的容量。有界缓冲区的 API 规定如下：
    • read(buf, sz):
      • 从有界缓冲区中读取最多 sz 个字符到 buf 中
      • 从有界缓冲区中删除已读取的字符
      • 如果有界缓冲区为空，则应阻塞直到它变为非空或关闭
      • 返回读取的字符数
    • write(buf, sz):
      • 从 buf 写入 sz 个字符到有界缓冲区的末尾
      • 写入最多 CAP 个字符到有界缓冲区的容量
      • 如果有界缓冲区变满，它应该阻塞直到缓冲区变为非满状态
      • 返回写入的字符数

    未实现同步的缓冲区¶

    struct bbuffer {
    static constexpr size_t bcapacity = 128;
    char bbuf_[bcapacity];
    size_t bpos_ = 0;
    size_t blen_ = 0;
    bool write_closed_ = false;

    ssize_t read(char* buf, size_t sz);
    ssize_t write(const char* buf, size_t sz);
    void shutdown_write();
};

ssize_t bbuffer::write(const char* buf, size_t sz) {
    assert(!this->write_closed_);
    size_t pos = 0;
    while (pos < sz && this->blen_ < bcapacity) {
        size_t bindex = (this->bpos_ + this->blen_) % bcapacity;
        size_t bspace = std::min(bcapacity - bindex, bcapacity - this->blen_);
        size_t n = std::min(sz - pos, bspace);
        memcpy(&this->bbuf_[bindex], &buf[pos], n);
        this->blen_ += n;
        pos += n;
    }
    if (pos == 0 && sz > 0) {
        return -1;  // try again
    } else {
        return pos;
    }
}

ssize_t bbuffer::read(char* buf, size_t sz) {
    size_t pos = 0;
    while (pos < sz && this->blen_ > 0) {
        size_t bspace = std::min(this->blen_, bcapacity - this->bpos_);
        size_t n = std::min(sz - pos, bspace);
        memcpy(&buf[pos], &this->bbuf_[this->bpos_], n);
        this->bpos_ = (this->bpos_ + n) % bcapacity;
        this->blen_ -= n;
        pos += n;
    }
    if (pos == 0 && sz > 0 && !this->write_closed_) {
        return -1;  // try again
    } else {
        return pos;
    }
}

    当我们尝试多个线程访问缓冲区的时候，并不能达到目的！原因是有关有界缓冲区的内部状态没有同步。 bbuffer::read() 和 bbuffer::write() 都修改了 bbuffer 对象的内部状态（尤其是 bpos_ 和 blen_ ），这样的访问在多线程环境中需要同步才能正确工作 - 回想一下同步的基本规则，即如果状态被多个线程访问，并且至少有一个线程可能对状态进行写入，则需要同步。

    :::success 修复有界缓冲区的一种方法是使用互斥锁将 read() 和 write() 方法的函数体转换为临界区。

    :::

    正确同步的缓冲区¶

    同步基本规则：如果两个或多个线程可以同时访问一个对象，并且至少有一个访问是写操作，则可能发生竞态条件，需要进行同步。

    有界缓冲区的内部状态， bbuf_ ， bpos_ ， blen_ 和 write_closed_ 都被 read() 和 write() 方法修改和读取。在这些方法中定义的局部变量不是共享的。我们需要对共享变量（缓冲区的内部状态）进行同步，而不是对局部变量进行同步。

    struct bbuffer {
    ...
    std::mutex mutex_;
    ...
};

ssize_t bbuffer::write(const char* buf, size_t sz) {
    this->mutex_.lock();
    ...
    this->mutex_.unlock();
    if (pos == 0 && sz > 0) {
        return -1;  // try again
    } else {
        return pos;
    }
}

ssize_t bbuffer::read(char* buf, size_t sz) {
    this->mutex_.lock();
    ...
    if (pos == 0 && sz > 0 && !this->write_closed_) {
        this->mutex_.unlock();
        return -1;  // try again
    } else {
        this->mutex_.unlock();
        return pos;
    }
}

    以上代码正确地实现了一个同步的有界缓冲区。简单地使用互斥锁将对共享状态的访问包装在临界区内，是实现复杂的内存对象同步（或者说“线程安全”）的最简单、也可能是最常见的方法。

    :::color3 但在多线程编程中，使用互斥锁来同步线程操作有不同策略：细粒度同步和粗粒度同步

    细粒度同步

    • 定义: 细粒度同步指的是为每个共享资源（例如，每个有界缓冲区）分别使用单独的互斥锁。
    • 实现: 每个有界缓冲区对象都有自己的互斥锁，线程在操作某个特定缓冲区时只会锁定这个缓冲区的互斥锁。
    • 优点: 允许线程并行操作不同的有界缓冲区，提高了并发性。例如，线程A操作缓冲区1时，线程B可以同时操作缓冲区2，互不影响。
    • 缺点: 需要管理更多的互斥锁，增加了代码的复杂性。

    粗粒度同步¶

    • 定义: 粗粒度同步指的是使用一个全局的互斥锁来保护所有的共享资源。
    • 实现: 所有的有界缓冲区共享一个全局互斥锁，线程在操作任何缓冲区时都会锁定这个全局互斥锁。
    • 优点: 实现简单，只需管理一个互斥锁。
    • 缺点: 同时限制了并发性。例如，线程A在操作缓冲区1时，线程B无法操作缓冲区2，直到线程A释放锁。这种同步方式会造成线程间的等待，从而降低系统的并发性能。

    :::

    C++ Mutex Patterns C++互斥模式¶

    我们经常会写一些需要先锁定互斥锁、做一些工作，然后在函数返回之前解锁互斥锁的同步函数。

    重复这样做可能会很繁琐。并且如果函数需要在多个地方选择返回，有可能忘记在返回之前加上解锁语句，导致错误。C++ 有一个模式可以帮助我们解决这些问题并简化编程：作用域锁。

    我们使用作用域锁来简化 bbuffer-scoped.cc 中有界缓冲区的编程。现在， write() 方法看起来如下所示：

    ssize_t bbuffer::write(const char* buf, size_t sz) {
    std::unique_lock guard(this->mutex_);
    assert(!this->write_closed_);
    size_t pos = 0;
    while (pos < sz && this->blen_ < bcapacity) {
        size_t bindex = (this->bpos_ + this->blen_) % bcapacity;
        this->bbuf_[bindex] = buf[pos];
        ++this->blen_;
        ++pos;
    }
}

    注意，在函数体的第一行中，我们声明了一个 std::unique_lock 对象，它是一个作用域锁，用于在函数的作用域内锁定互斥量。在 std::unique_lock 对象初始化时，互斥量会自动锁定，当该对象超出作用域时，互斥量会自动解锁。这些特殊的作用域锁对象在它们的构造函数和析构函数中（一个特殊的 C++类方法，在销毁该类的对象之前被调用）锁定和解锁互斥量，以实现这种效果。

    :::success 这种设计模式也被称为资源获取即初始化（Resource Acquisition is Initialization）（RAII），在软件工程中是一种常见的模式。使用 RAII 可以简化编码，同时避免某些编程错误。

    :::
