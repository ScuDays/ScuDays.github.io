    Lecture 24：条件变量、总结¶

    条件变量使得线程能够在条件满足时实现阻塞。这提高了有界缓冲区程序的效率，因为线程不再自旋；对于其他需要线程等待的程序，为确保程序的正确性条件变量是必需的。

    为了等待资源，线程在持有互斥锁的情况下调用条件变量的 wait() 。如果条件不成立，互斥锁将被释放，线程将被阻塞。通过从另一个线程调用 notify_all （或 notify_one ），可以解除这个线程对条件变量的等待。

    我们通过考虑条件变量 API 的两个细节来结束我们对线程同步的讨论：为什么需要原子释放锁并阻塞调用线程，以及为什么需要将对其的调用包装在循环中。对于这两个选择，事实证明，如果没有它们，条件变量允许在多个线程以最不理想的方式交错时出现微妙的错误执行。

    条件变量¶

    条件变量是另一种同步对象。当程序中的一个线程持有锁但无法继续执行，持续等待某个资源成为真才能继续执行时，条件变量就非常有用。例如，即工作线程必须等待客户端处理线程插入工作或非满。

    重要的是，等待线程必须放弃其持有的锁，以便其他线程可以更改条件。这就是条件变量的作用。它支持以下操作：

    1. wait(std::unique_lock& lock) ：在一个原子步骤中，它解锁所持有的锁，并阻塞，直到另一个线程调用 notify_all()来唤醒它 。在被返回（唤醒）时，它还会尝试重新去锁定锁。

    2. notify_all() ：唤醒所有被调用 wait() 阻塞的线程。

    3. 来看例子：

    从逻辑上讲，当有界缓冲区变满时，写入者应该阻塞，并在缓冲区再次变为非满时解除阻塞。

    • 让我们在互斥锁下方的有界缓冲区中创建一个条件变量，称为 nonfull_ 。请注意，我们方便地将条件变量命名为函数应解除阻塞的条件。这将使代码在以后更容易阅读。 write() 方法实现阻塞的是 bbuffer-cond.cc 。
    ssize_t bbuffer::write(const char* buf, size_t sz) {
    std::unique_lock guard(this->mutex_);    //C++互斥模式 RAII
    assert(!this->write_closed_);
    while (this->blen_ == bcapacity) {  // #1
        this->nonfull_.wait(guard);
    }
    size_t pos = 0;
    while (pos < sz && this->blen_ < bcapacity) {
        size_t bindex = (this->bpos_ + this->blen_) % bcapacity;
        this->bbuf_[bindex] = buf[pos];
        ++this->blen_;
        ++pos;
    }
    • 当我们所做的一些更改可能使得解除阻塞条件时，应该调用 notify_all() 。在我们的场景中，这意味着我们必须在 read() 方法中调用 notify_all() ，该方法从缓冲区中取出字符并有可能解除写入器的阻塞。
    ssize_t bbuffer::read(char* buf, size_t sz) {
    std::unique_lock guard(this->mutex_);
    ...
    while (pos < sz && this->blen_ > 0) {
        buf[pos] = this->bbuf_[this->bpos_];
        this->bpos_ = (this->bpos_ + 1) % bcapacity;
        --this->blen_;
        ++pos;
    }
    if (pos > 0) {                   // #2
        this->nonfull_.notify_all();
    }

    :::tips 使用条件变量，我们的有界缓冲程序运行效率显著提高：不再进行数百万次 read 和 write 的调用，而是进行约 10 万次 read 的调用和约 100 万次 write 的调用，因为线程在缓冲区满（写入）或空（读取）时被阻塞。

    :::

    • 为什么要在 wait() 周围使用 while 循环？

    由于被称为虚假唤醒（spurious wakeups）的原因，我们总是在循环中使用 wait() 。

    1. 由于 notify_all() 唤醒了在某个 wait() 调用上阻塞的所有线程，因此当一个特定的阻塞线程锁定互斥锁并开始运行时，可能已经有其他阻塞线程解除了阻塞，取得了一些进展，并将解除阻塞的条件改回了 false。
    2. 因此，”被唤醒”的线程在继续之前必须重新验证解除阻塞的条件，如果条件不满足，则必须返回阻塞状态。 while 循环正是实现了这一点。

    否则，可能又会产生两个或多个线程被同时唤醒，并同时获取了互斥锁，访问了共享资源的情况，这是不符合我们设想的同步的。
