    Lecture 22：同步、原子操作和互斥锁 Synchronization, Atomics, and Mutexes¶

    讨论了如何避免程序中的竞态条件。为了做到这一点，我们添加了同步机制，通过不同线程之间的有序访问来防止它们以未定义的方式交错。同步最终依赖于处理器原子指令，并使用基于这些原子指令构建的抽象来实现它。 std::atomic （用于原始类型）和互斥锁（用于同步对更复杂数据结构的访问）是 C++程序可能用于同步多个线程访问的同步对象的方法。

    Atomics 原子操作¶

    以下代码使用 C++ 原子操作实现了同步的共享内存访问，编译器会将其转换为原子指令。

    void threadfunc(std::atomic* x) {
    for (int i = 0; i != 10000000; ++i) {
        x->fetch_add(1);
        // `*x += 1` and `(*x)++` also work!
    }
}
    • C++ 的原子操作库使用 x86-64 的原子指令实现原子加法。当我们使用 objdump 来查看 threadfunc() 的汇编代码时，我们会看到一个 lock addl … 指令，而不仅仅是 addl … 。 addl 指令的 lock 前缀要求处理器在整个 addl 指令完成之前保持对共享变量的缓存行的控制（或者按照 Intel 的术语来说，”锁定内存总线”）。
    • C++原子操作和以 lock 为前缀的指令只适用于不能跨越多个缓存行的字大小变量。（回想一下，对齐单元的讨论中提到对齐的一个原因是为了避免原始类型跨越缓存块；在这里，这个概念再次出现！）

    什么是缓存行？

    存行是 CPU 缓存中的一个存储单位，CPU 每次从主存中读取数据时，会一次性读取一个缓存行的数据。

    什么是对齐单位？

    “对齐单位”是指数据在内存中的分布必须遵循某种特定规则，这通常是为了提高数据访问的效率。例如，如果一个整数变量跨越了两个缓存行，那么每次访问这个变量时，CPU 都需要读取两个缓存行，这会降低数据访问的效率。

    Synchronization Objects 同步对象¶

    • 通过使用 C++ 的 std::atomic 模板库来实现原始类型（如整数）的同步，该库会自动将用于执行更新的 addl 指令转换为以 lock 为前缀的指令，以实现原子操作。
    • 但std::atomic 仅适用于机器字长或更小的整数。要在内存中同步更复杂的对象或执行更复杂的同步任务，我们仍然需要使用称为同步对象的抽象。

    Mutexes 互斥锁¶

    • 互斥意味着最多只有一个线程可以同时访问共享数据。

    在上一堂课的多线程 incr-basic.cc 示例中，代码无法正常工作，因为多个线程可以同时访问共享变量。如果强制执行互斥策略，代码将会正确运行。我们可以使用一个 mutex 对象来强制执行互斥（ incr-mutex.cc ）。在这个示例中，它的效果与将 *x 包装在 std::atomic 模板中相同（上面原子操作中的示例）。

    std::mutex mutex;
void threadfunc(unsigned* x) {
    for (int i = 0; i != 10000000; ++i) {
        mutex.lock();
        *x += 1;
        mutex.unlock();
    }
}

    :::warning + 互斥锁（一种锁数据结构）具有内部状态（用 state 表示），可以是锁定或解锁的。互斥锁对象的语义如下： 1. 初始化时， state = unlocked 。 2. mutex::lock() ：等待 state 变为 unlocked ，然后原子性地设置为 state = locked 。请注意，这两个步骤应在一个原子操作中完成。 3. mutex::unlock() ：断言 state == locked ，然后设置为 state = unlocked 。

    互斥策略在 lock() 和 unlock() 调用之间的代码区域中执行。我们将这个区域称为临界区 (critical section)。

    :::

    互斥锁确保在临界区内只有一个线程可以活动。换句话说，互斥锁强制执行线程交错。

    就像下图所示：T1 在等待互斥锁 m ，而 T2 在临界区内运行，然后 T1 在 T2 通过 m.unlock() 释放互斥锁后获得互斥锁后进入临界区。

    原子操作如何进入互斥锁？考虑 m.lock() 的实现：

    必须有一个操作允许调用 lock() 的线程将锁的状态设置为 locked ，且同时不会有其他线程抢占锁的机会。

    我们确保这一点的方式是使用原子操作：特别是，一个”比较并交换”操作（在 x86-64 中为 CMPXCHG）原子地读取锁的状态，并且如果它是 unlocked ，则将其设置为 locked 。

    使用单个位和 cmpxchg 实现互斥锁
    在内部，互斥锁可以使用原子计数器实现，或者使用单个原子位！但需要一些特殊的原子机器指令。 忙等待互斥锁（也称为自旋锁）可以如下实现：
    struct mutex {
    std::atomic spinlock;
    void lock() {
        while (spinlock.swap(1) == 1) {}
    }
    void unlock() {
        spinlock.store(0);
    }
};
    1. spinlock.swap() 方法是一个原子交换方法，它在一个原子步骤中将指定的值存储到原子 spinlock 变量中，并返回变量的旧值。
    2. 它之所以有效，是因为除非 spinlock 先前包含值 0（表示未锁定），否则 lock() 不会返回。在这种情况下，它将原子地将值 1（表示已锁定）存储到 spinlock 中，并阻止其他 lock() 调用返回，从而确保互斥。在自旋等待期间，它只是将 spinlock 的旧值 1 与 1 进行交换，实际上不改变锁的状态。请花点时间欣赏这个简单的构造如何正确实现互斥。
    • x86-64 通过 lock xchg 汇编指令提供了原子交换功能。这就是上面实现互斥锁（）的关键。
    • x86-64 还提供了一种更强大的原子指令，进一步打开了在现代处理器中可以原子操作的可能性。该指令称为比较交换，或 lock cmpxchg 。它足够强大，可以实现原子交换、加法、减法、乘法、平方根等许多其他操作。

    该指令的行为定义如下：

    // Everything in one atomic step
int compare_exchange(int* object, int expected, int desired) {
    if (*object == expected) {
        *object = desired;
        return expected;
    } else {
        return *object;
    }
}

    该指令也可以作为 C++ std::atomic 类型对象的 this->compare_exchange_strong() 成员方法进行访问。它不返回旧值，而是返回一个布尔值，指示交换是否成功。
