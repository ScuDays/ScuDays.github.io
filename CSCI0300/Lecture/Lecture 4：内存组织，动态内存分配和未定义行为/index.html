    Lecture 4：内存组织，动态内存分配和未定义行为

    探讨了变量的生命周期以及计算机程序中不同生命周期变量的不同内存段：静态段（具有静态生命周期）、栈（具有自动生命周期）和堆（具有动态、由程序员管理的生命周期）。了解了可能的新分配的未初始化内存的内容。

    • 程序及其数据只是内存中的字节
    • 我们需要详细了解数据在内存中的表示方式。

    为什么？

    • 现在正在建立对C/C++程序（实际上，也包括其他语言的程序！）不同部分在内存中存储的理解。这将帮助理解程序如何获取和管理内存，一些编程语言（例如Java、OCaml、Pyret）会在您不知情的情况下自动完成这些操作，而其他编程语言，特别是C和C++这样的系统编程语言，会强制您作为程序员来管理一部分内存。这给您带来了很大的控制权，并且可以避免昂贵的隐藏内存分配和复制成本。
    char global_ch = 'A';   //全局变量
const char const_global_ch = 'B'; //全局常量
void f() {
    char local_ch = 'C';            //局部变量
    hexdump(&global_ch, 1);
    hexdump(&const_global_ch, 1);
    hexdump(&local_ch, 1);
}

    hexdump()打印出地址，及地址的内容 。

    这里存在两种变量生命周期：

    1. 全局变量 global_ch 和 const_global_ch 需要在程序的整个运行时间内存在，因为程序可能在代码的任何地方引用它们。这被称为静态生命周期。
    2. 本地变量 local_ch 需要保持到它超出作用域，即当执行到 f() 的闭合大括号时。超出作用域后，程序中的任何代码都无法引用该变量，因此可以重新使用其内存。这被称为自动生命周期。本地变量和函数参数具有自动生命周期。

    但是如果一个函数需要创建一个__在程序开始时大小未知（因此不能是全局的）__并且需要在函数结束后继续存在的对象呢？在这种常见情况下，静态生命周期和自动生命周期都不合适。

    • Dynamically Allocated Objects 动态分配对象

    为了实现这个目的，C语言允许第三种对象生命周期：动态生命周期。

    1. 对于具有动态生命周期的对象，必须显式地创建和销毁对象 - 也就是说，必须为对象分配内存，并确保在对象不再需要时释放它。

    为类型指针分配内存，并赋值给类型指针所指的内容。

    void f() {
    char* allocated_ch = malloc(1);
    *allocated_ch = 'D';
}

    与Java的相似之处和不同之处

    调用 malloc() 可能看起来笨拙，但它们实际上与Java中的 new 关键字做的事情是一样的：为一个新对象分配内存。事实上，C++实际上提供了一个 new 关键字，在底层调用 malloc() 。然而，与Java相比，一个很大的区别是你需要负责清理和返回那块内存。Java会自动判断何时不再需要具有动态生命周期的对象，并在那时释放内存（这个过程称为“垃圾回收”）。C和C++不会这样做，而是将决定何时返回内存的时机留给程序员。

    1. 我们需要释放内存，通过调用 free() 函数并将分配的盒子的地址作为参数来实现这一点。例如， free(allocated_ch); 将释放我们要求 malloc() 为 allocated_ch 预留的内存。

    动态生命周期的错误使用是C/C++程序中问题、错误和安全漏洞的极为常见的来源：严重的问题，如内存泄漏、双重释放、使用已释放的内存等，都源于这种语言特性。

    • Memory Segments 内存段

    具有不同生命周期的对象被分组存储在内存的不同区域中。程序代码、全局变量和常量全局变量都存储在静态段中，因为它们在编译时具有静态生命周期和已知大小。

    1. 我们查看由 mexplore 打印的地址，我们可以看到存储在其中的全局变量位于相对较低的内存地址（大约 0x60‘0000 十六进制和 0x40‘0000 十六进制），而局部变量存储在一个较高的地址，接近 0x7fff’ffff’ffff （大约2^47 ），而动态分配的字符存储在两者之间（尽管更接近静态段）。

    2. 这种安排有其原因：它允许两种类型的段都能够自由增长，而不会相互干扰。特别是，自动生命周期段随着更多的局部变量进入作用域而向下增长，而动态生命周期段则向上增长。如果它们相遇，你的计算机将会耗尽内存。

    3. 自动生命周期段被称为栈(就是堆栈)。它之所以被称为栈，是因为它像一叠纸一样增长和缩小：当你调用一个函数时，它的自动生命周期对象被添加到现有的自动对象的左侧，当函数返回时，程序再次释放它们的内存。

    4. 动态生命周期段被称为堆，它通常在内存地址上向上增长。但与栈不同，它可以有“空洞”：如果程序员销毁了位于内存中两个对象之间的对象，那么之间就会有未使用的内存（空隙）。
    5. 静态生命周期段中保存的内容
    6. 保存程序被称为“文本段””text segment” ——因为它保存程序文本；尽管“文本”是计算机语言；
    7. 只读全局变量的部分 read-only globals (“rodata”) 例如：常量字符串
    8. 可修改全局变量的部分（modifyable globals (“data” and “bss”)。

    Java Similarities Note Java相似性注释

    在Java中，使用 new 关键字创建的任何对象都具有动态生命周期，并存储在堆上。（Java只将“原始”类型（ int ， double 等）存储在栈上。）实际上，Java在底层使用 malloc() ！然而，似乎Java对所有对象都具有自动生命周期，因为您从不需要显式地销毁它们！这是因为您的程序在Java虚拟机（JVM）内运行，JVM会“神奇地”注入代码来跟踪每个对象是否仍然可以通过作用域内的变量访问；如果引用计数变为零，JVM会自动调用 free() 来删除对象。但是，所有这些魔法在性能方面并非免费，因为需要运行代码来跟踪对象的引用计数。相比之下，C和C++选择什么都不做，并为程序员提供最大的控制权，无论是好是坏。

    • Example: Strings of dynamic length 动态长度的字符串
    • 如何为一个动态字符串分配内存？

    答案是我们在动态生命周期段中分配了正确数量的字节。这就是它获取内存的方式。

    1. 我们能否将一个字符串直接赋值给它？不行，因为 allocated_st 的类型是 char ，一个指向 char （具体来说是字符串中的第一个字符）的指针。*

    • Uninitialized Memory 未初始化的内存

    malloc()__ 向操作系统请求一些动态段的内存，并返回指向新分配内存的第一个字节的指针。但是这块内存的内容是什么呢？

    void f() {
    char* allocated_st1 = malloc(100);
    char* allocated_st2 = malloc(100);
    sprintf(allocated_st1, "C programming is cool");
    sprintf(allocated_st2, "Strings are sequences of bytes");

    hexdump(allocated_st1, 100);
    hexdump(allocated_st2, 100);

    free(allocated_st1);
    free(allocated_st2);

    char* allocated_again_st = malloc(100);
    sprintf(allocated_again_st, "C programming is cool");
    hexdump(allocated_again_st, 100);
}
    1. 它两次分配了100字节的动态生命周期内存；
    2. 然后将两个字符串写入该内存，并使用 hexdump() 打印它们；
    3. 然后使用 free() 释放内存，因此这200字节内存现在是未分配的；
    4. 最后，程序再次请求100个字节，可能是为了另一个字符串，并打印出该内存的内容。

    结果：发现最后一次中内存中存在有前面两次分配中残留的信息。

    我们的程序非常快速：计算机不需要对释放后得到的新内存做任何工作，只需直接使用它。

    但这非常危险，因为很容易通过未初始化的内存泄露秘密信息。

    想象一下，你将社会安全号码和信用卡详细信息存储在第一个字符串中，然后使用 free() 释放它，然后程序的其他部分（可能由邪恶的人控制）请求内存并获取这些字节。这种情况在现实世界中经常发生，是计算机安全问题的重要来源。

    How do you avoid leaking secrets through uninitialized memory?
    如何避免通过未初始化的内存泄露机密信息？
    为了避免程序意外泄露信息，作为程序员，在调用 free() 之前必须覆盖内存。这在处理加密信息和密码的应用程序中非常常见：在使用 free() 释放内存之前，它们会在内存中写入零。
