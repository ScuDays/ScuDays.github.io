    Lecture 19：进程间通信 ¶

    • 我们通过引入进程间通信抽象机制来打破进程的绝对隔离。一个简单的、一次性的通信机制是父进程在等待子进程退出后使用 waitpid() 读取的退出状态。由于这个退出状态需要在子进程退出后仍然可用，操作系统内核会将进程保留为 " 僵尸进程 "，直到父进程调用 waitpid() 并检索（或忽略）退出状态。
    • 随后深入探讨进程如何通过管道进行交互，管道是内核介导的共享内存缓冲区，进程通过文件描述符访问它们。我们看到，将管道与分叉结合使用可以使进程与其子进程（甚至是子进程之间）建立通信通道。这是一个非常强大的抽象，它使我们能够链接 shell 命令，并在多处理器计算机上使用多个进程实现并行处理。

    进程间通信 Inter-Process Communication - IPC¶

    我们引入虚拟内存来确保进程的隔离，但有时进程之间需要传递信息。

    例如，通过多个进程分担工作来加速大型任务，子进程需要将结果反馈给父进程。但是，严格的隔离限制了这种可能性。

    幸运的是，有一系列的机制可以让进程以安全的方式相互通信，通常由内核进行中介。这些通信方式和实现它们的抽象被称为进程间通信。

    最简单的 IPC：wait 等待进程退出 ¶

    等待进程退出：waitpid() ¶

    • 最简单的 IPC 形式是进程的退出状态，它给予进程一次性机会向父进程发送一个整数。

    Linux 上的进程通常使用这个整数来指示它们是否成功退出（零退出状态）或是否发生错误（非零退出状态，通常是负数）。

    我们通过wait 系列系统调用来获得子进程的返回代码。我们特别关注其中一种：waitpid() 系统调用。

    • waitpid() 有两个作用 1. 它允许父进程等待子进程退出。这在某些情况下非常有用，例如父进程希望子进程先执行完再执行。 2. waitpid() 允许父进程读取子进程的退出状态。

    来看例子：

    • 创建一个子进程。
    • 子进程睡眠半秒钟，打印出一条消息，然后退出。
    • 父进程等待子进程完成，并根据子进程的退出状态打印出一条消息。

    需要注意的是 : 我们必须在 父进程中调用 waitpid() 。

    waitpid() 的第一个参数是要等待的进程的 PID。这必须是当前进程的直接子进程！进程不能等待不是它的子进程的 PID - 如果尝试这样做， waitpid() 将返回一个错误。

    第二个参数是一个指向整数的指针，内核将在其中存储子进程的退出状态。

    int main() {
    fprintf(stderr, "Hello from parent pid %d\n", getpid());

    // Start a child
    pid_t p1 = fork();
    assert(p1 >= 0);
    if (p1 == 0) {
        usleep(500000);
        fprintf(stderr, "Goodbye from child pid %d\n", getpid());
        exit(0);
    }
    double start_time = tstamp();

    // Wait for the child and print its status
    int status;
    pid_t exited_pid = waitpid(p1, &status, 0);
    assert(exited_pid == p1);

    if (WIFEXITED(status)) {
        fprintf(stderr, "Child exited with status %d after %g sec\n",
                WEXITSTATUS(status), tstamp() - start_time);
    } else {
        fprintf(stderr, "Child exited abnormally [%x]\n", status);
    }
}

    Blocking vs. polling 阻塞 vs. 轮询
    阻塞与轮询相比，可以更高效地以编程方式 " 等待事件发生 "。这是我们在课程中会反复看到的一种范式，它依赖于用户空间和内核之间的协作。具体来说，许多系统调用会 " 阻塞 "，直到特定条件为真，而内核的工作是不调度被阻塞的进程。

    僵尸进程 ¶

    • 当一个进程 fork 出一个子进程时，子进程最终会退出并具有一个退出状态——为了 waitpid() 的调用，这个退出状态需要存储在内存中的某个地方，但子进程已经退出，所以子进程的地址空间已经被销毁了！所以跟踪退出状态的责任落在内核上，这意味着它需要保留有关已退出子进程的信息，直到父进程调用 waitpid() （或自身退出）。
    • 因此，子进程在退出后进入“僵尸状态”：进程已经退出，但内核仍然保留其 PID 和退出状态，等待进程上调用 waitpid() 。僵尸进程会消耗内核资源，我们应尽量避免出现僵尸进程！
    • 避免僵尸进程的技巧是至少为每个子进程调用 waitpid() 一次。使用 -1 作为 pid 参数调用 waitpid() 将检查任意子进程的退出状态。

    更强大的 IPC：Pipes （管道）¶

    介绍 ¶

    我们已经在上次看到了进程退出状态的简单示例中使用了 IPC。

    此外，Linux 还提供了其他进程通信的机制：它们可以在磁盘上读写文件，或者可以依赖于更高效的内存流传输，即管道。

    多个进程共同合作以实现目标的概念在日常计算中以多种方式出现。以下有两个示例：

    • 通过 | （" 管道 "）字符组合多个 Shell 命令：例如 wc -l .cc | sort -n ，它计算当前目录中所有 C++ 源文件的行数（ wc -l .cc ），然后使用另一个命令（ sort -n ）按行数对结果进行排序。这里的管道符号告诉 Shell 在它们之间建立一个管道，并要求第一个命令将其输出发送到管道，然后第二个命令从管道中读取。
    • 加快与大数据集或多个请求交互的操作，通过将工作分配给与父进程通信结果的多个进程。如果您使用具有多个处理器的计算机，这种方法尤其有效，在最佳情况下，N 个进程可以实现单个进程的 N 倍速度或吞吐量。

    Pipe Setup 管道设置 ¶

    • 管道提供了一种单向、有序的传输机制。进程可以使用管道将数据发送给自己，但是当与 fork() 结合使用时，管道在父子进程之间的通信中最为强大。
    • 使用 pipe() 系统调用创建管道。每个管道都有两个面向用户的文件描述符，分别对应管道的读端和写端。文件描述符是内核用来跟踪用户空间进程使用的打开资源（如文件）的标识符。用户空间进程使用整数文件描述符（FD）号引用这些资源；在内核中，FD 号索引到每个进程维护的 FD 表中。

    pipe() 系统调用的签名如下：

    C
    1
2
    //传递一个2空间的数组来进行存储信息。
int pipe(int pfd[2]);

    用户空间进程有责任为 pfd 数组分配内存 —— 用来存储有关 描述符 等信息

    内存可以是全局变量、堆栈或堆上的。这遵循 Linux 系统调用的一般原则：内核写入数据或信息的内存必须由用户空间分配，并传递一个有效的指针给内核。

    成功调用 pipe() 会创建 2 个文件描述符，放置在数组 pfd 中：

    • pfd[0]: 读取管道的末尾
    • pfd[1]: 写入管道的末尾

    写入到 pfd[1] 的数据可以从 pfd[0] 读取。

    • 如何记住哪一端是哪一端？ 1. 每个进程可用的默认文件描述符（ stdin ， stdout 和 stderr ）为记住管道的读取端提供了一个有用的助记符： 2. 0 是 stdin 的 FD 编号，1 是 stdout 的 FD 编号 3. 从标准输入读取程序，并写入标准输出 4. pfd[0] 是读取端（输入端）， pfd[1] 是写入端（输出端）

    • 管道可以双向使用吗？

    管道的读端不能写入，写端不能读取。尝试读 / 写错误的管道端口将导致系统调用错误（ read() 或 write() 调用将返回 -1）。

    单个进程中的管道 ¶

    来看一个具体例子：

    • 代码在一个进程中创建了一个管道，向其写入数据，并从中读取数据。
    • 但在 write() 系统调用完成后， read() 从管道中调用之前，数据去了哪里？

    数据并不驻留在进程的地址空间中！实际上，它进入了位于内核地址空间中的内存缓冲区。

    int main() {
    int pfd[2]; // 用于存储管道的文件描述符
    int r = pipe(pfd); // 创建一个管道，pfd[0]为读取端，pfd[1]为写入端
    assert(r == 0); // 确保管道创建成功

    char wbuf[BUFSIZ]; // 写入缓冲区
    sprintf(wbuf, "Hello from pid %d\n", getpid()); // 将当前进程的PID写入缓冲区

    ssize_t n = write(pfd[1], wbuf, strlen(wbuf)); // 将缓冲区内容写入管道
    assert(n == (ssize_t) strlen(wbuf)); // 确保写入的字节数与缓冲区的长度相同

    char rbuf[BUFSIZ]; // 读取缓冲区
    n = read(pfd[0], rbuf, BUFSIZ); // 从管道读取数据到缓冲区
    assert(n >= 0); // 确保读取成功
    rbuf[n] = 0; // 在读取的末尾加上一个空字符，形成字符串

    assert(strcmp(wbuf, rbuf) == 0); // 确保写入的内容和读取的内容相同
    printf("Wrote %s", wbuf); // 打印写入的内容
    printf("Read %s", rbuf); // 打印读取的内容
}

    Pipes across processes 多个进程间的管道 ¶

    我们刚刚看到了在同一进程中使用管道的功能。

    由于管道存在于内核中，它也可以用于在进程之间传递数据。我们可以将 fork() 系统调用（将父进程的打开文件描述符复制到子进程）与管道结合起来，建立跨进程的通信通道。

    来看例子：

    在使用 fork() 来创建一个子进程，在 fork 之前我们首先创建了一个管道！ fork() 在子进程中复制了两个管道文件描述符，但请注意，管道本身没有被复制（因为管道不在进程的地址空间中）。子进程然后向管道写入一条消息，父进程可以从中读取相同的消息。实现了进程间通信！

    int main() {
    int pipefd[2]; // 用于存储管道的文件描述符
    int r = pipe(pipefd); // 创建一个管道，pipefd[0] 是读取端，pipefd[1] 是写入端
    assert(r == 0); // 确保管道创建成功

    pid_t p1 = fork(); // 创建一个子进程
    assert(p1 >= 0); // 确保fork调用成功

    if (p1 == 0) {
        // 子进程
        const char* message = "Hello, mama!"; // 子进程要发送的消息
        printf("[child: %d] sending message\n", getpid()); // 打印子进程发送消息的信息
        ssize_t nw = write(pipefd[1], message, strlen(message)); // 子进程将消息写入管道
        assert(nw == (ssize_t) strlen(message)); // 确保写入的字节数与消息的长度相同
        exit(0); // 子进程退出
    }

    // 父进程
    char buf[BUFSIZ]; // 读取缓冲区
    if (read(pipefd[0], buf, BUFSIZ) > 0) { // 父进程从管道读取数据
        printf("[parent: %d] I got a message! It was “%s”\n", getpid(), buf); // 打印读取到的消息
    }
    close(pipefd[0]); // 关闭管道的读取端 
    close(pipefd[1]); // 关闭管道的写入端
}
    • 请注意，在上述情况中，我们有四个与管道相关联的文件描述符，因为 fork() 复制了与管道两端对应的文件描述符。在这种情况下，管道有两个读端和两个写端。下面的动画图片以维护两个进程的 FD 表的方式说明了这种情况。

    文件描述符表是每个进程在内核中维护的一张表，记录了该进程打开的所有文件、管道、套接字等的文件描述符。

    • 请注意，在子进程中仍然存在一个管道的写入端和一个读取端，它们从未关闭！ 1. 我们只在父线程中关闭了 pipefd[0] 和 pipefd[1] ，

    这可能会导致在使用管道时出现奇怪的挂起情况，因此一个常见的习惯用法是在分叉后立即关闭进程不需要的管道端口。在这个例子中，我们需要关闭子进程中的写入和读取端口。
