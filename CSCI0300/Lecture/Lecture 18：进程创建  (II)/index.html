    Lecture 18：进程创建 (II) ¶

    1. 如何运行不同的程序呢？：如果我们只有 fork()，我们只能执行单个用户空间进程的副本。实际我们希望能够从用户空间进程启动其他程序。
    2. UNIX 的解决方案：fork-and-exec style：在 UNIX 中有一系列的系统调用可以执行新的程序。我们讨论 execv() 系统调用（以及 exec 家族中的所有系统调用）执行操作：清除当前进程的虚拟地址空间，开始在当前进程中执行指定的程序，从其 main() 函数开始。
    3. fork-and-exec：先 fork 当前进程，然后在新的进程中 execv(）得到一个运行指定程序的子线程。
    4. 替代接口：posix_spawn()。它在不复制地址空间或销毁当前进程的情况下创建一个新进程。
    5. 为什么还要用 fork()？：哪种进程创建方式更好的争论从未停止。现代 UNIX 操作系统从 1970 年代的原始 UNIX 继承了 fork-and-exec 风格，其中 fork() 的实现非常容易。现代 UNIX 系统可以非常有效地执行 fork()，在必要之前不实际执行任何实质性的复制（使用写时复制优化）。
    6. 不用 fork() 直接运行 execv()：execv() 删除了旧进程的映像（包括代码、全局变量、堆和栈），没有新的进程被创建。新的程序在旧的程序被销毁后在同一个进程中运行。

    如何运行不同的程序¶

    如果我们只有 fork()，我们只能执行单个用户空间进程的副本。实际我们希望能够从用户空间进程启动其他程序。

    UNIX 下解决方案 ¶

    • UNIX 方式： fork -and- exec style

    execv() 系统调用（以及 exec 系列的所有系统调用）执行以下操作：

    1. 清除当前进程的虚拟地址空间。
    2. 从其 main() 函数开始，在当前进程中执行指定的程序。

    注意：

    1. execv 不会“生成”一个进程。它会销毁当前进程并替换它。因此，通常会与 fork 结合使用 execv ：首先调用 fork() 创建一个子进程，然后调用 execv() 在子进程中运行一个新程序，替换 fork() 复制的“进程镜像”。
    2. 在 execv 后面的代码都不会被执行，但如果 exec 失败，它会返回，我们不能编写假设它永远不会返回的代码。
    3. exec 系统调用会保留当前的文件描述符表单。所以任何在 exec 系统调用之前的文件描述符，例如 0，1，2 等。它们在新的程序中表示相同的东西。

    例子：

    我们现在将使用 execv() 系统调用来运行此程序。

    现代 Linux 系统下替代接口：posix_spawn()¶

    连续调用 fork() 和 execv() 来运行一个进程可能看起来有些违反直觉，甚至效率低下。

    想象一下一个复杂的程序，它有几个 GB 的虚拟地址空间映射，并且它想要创建一个新的进程。如果我们要做的只是扔掉一切重新开始，那么复制当前程序的大虚拟地址空间有什么意义呢？

    这些是关于 UNIX 风格进程管理的合理担忧。

    现代 Linux 系统提供了一种另外的系统调用，称为 posix_spawn() ，它可以在不复制地址空间或销毁当前进程的情况下创建一个新进程。一个新程序在一个新进程中被 " 生成 "，新进程的 pid 通过其中一个指针参数返回。像 Windows 这样的非 UNIX 操作系统也使用这种进程创建方式。

    int main() {
    const char* args[] = {
        "./myecho", // argv[0] is the string used to execute the program
        "Hello!",
        "Myecho should print these",
        "arguments.",
        nullptr
    };
    fprintf(stderr, "About to spawn myecho from pid %d\n", getpid());
    pid_t p;
    int r = posix_spawn(&p, "./myecho", nullptr, nullptr,
                        (char**) args, nullptr);
    assert(r == 0);
    fprintf(stderr, "Child pid %d should run myecho\n", p);
}

    请注意， posix_spawn() 需要比 execv() 更多的参数。这与管理新进程运行的环境有关。查看 posix_spawn 的手册页面，了解这些额外的 nullptr 参数是关于什么的 - 它们非常复杂。

    这教会我们一个有趣的 API 设计教训：在许多情况下，API 的性能和可用性往往是一种权衡。
