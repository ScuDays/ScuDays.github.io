    Lecture 15：特权分离，内存保护¶

    对操作系统内核如何与硬件协同工作并建立特权概念有了更好的理解：内核在特权模式（privileged mode）下使用处理器，而用户空间进程则不是。当用户空间进程尝试执行需要特权的操作时，会发生中断并将控制权交还给内核。通常情况下，内核会终止违规的进程！

    研究了硬件机制和操作系统内核如何协同工作，以确保即使程序出现问题，我们的计算机也能稳定运行。特别是，我们发现只有内核才能执行特权操作，如果用户空间进程试图执行这些操作，会导致故障(trap)并将CPU的控制交还给内核。这使得操作系统内核能够防止任何试图接管CPU的攻击，并确保用户空间进程以较弱权限运行。

    • Userspace/Kernel Privilege Separation 用户空间/内核特权分离

    共享硬件资源：大多数计算机硬件资源是共享的，内核通过时间分片（time-sharing）来管理这些资源的使用，即不同程序轮流使用处理器等资源。

    • Representing Privilege 表示特权：

    内核如何获得特权，并确保它比用户空间进程保持更高的特权？再次，硬件和操作系统之间的相互作用在这里起着关键作用。

    1. 内核是计算机启动时运行的第一个程序。当计算机启动（”引导”）时，它以完全特权（full privileged ）运行 - 换句话说，第一个运行的程序可以访问所有硬件，并读写任何内存位置。因此，第一个运行的程序最好是内核！然后，内核可以以受限特权启动其他程序。
    2. x86-64 计算机表示特权的一种方式是通过一个特殊的寄存器，即 %cs 寄存器。该寄存器可以存储值 0 或 3。0 表示处理器正在特权模式下运行（实际上意味着它正在运行内核），而 3 表示它在非特权模式下运行（通常是用户空间程序）。处理器的特权模式控制着当前进程是否可以执行关闭中断等操作。
    3. x86-64指令 cli （”清除中断”）的效果是关闭处理器中的所有中断处理。因此，如果一个进程能够运行这个指令，它可以忽略定时器中断。Eve可能会在她的进程中使用类似 asm volatile (“cli”); 的代码来尝试实现这一点，但是假设内核在启动用户空间进程时正确限制了权限，Eve尝试运行这个指令将导致异常（”fault”，更具体地说是x86-64术语中的一般保护故障），从而导致内核接管控制。

    当Eve试图执行特权指令时，内核应该做什么？由编写操作系统内核的程序员决定采用什么策略。

    例如，内核可以简单地忽略非法指令，并继续执行Eve的进程从下一条指令开始。或者内核可以试图惩罚Eve试图违反规则，并终止她的进程。以下代码添加到WeensyOS内核的异常处理程序中会终止Eve的进程。

    void exception(regstate* regs) {
    // [...]
    case INT_GP:
        if (regs->reg_cs & 3) {
          // Userspace fault, kill the process
          current->state = P_BROKEN;
          break;
        } else {
          goto unhandled;
        }
    // [...]
}

    在 WeensyOS 内核中， current 始终指向进入内核之前运行的最后一个用户空间进程的进程描述符（包含有关该进程的信息的结构）。

    因此， current->state 指的是当前进程的状态，并将其设置为 P_BROKEN 会将该进程标记为不可行，并阻止内核再次运行它。

    • Memory Protection 内存保护

    内核需要保护内存（RAM）免受用户空间进程的攻击，因为这些进程可能会修改内核代码以获得对计算机的控制。

    例如：如果允许Eve的进程写入内核内存，她可以通过修改内核代码内存将自己的无限循环移入内核。Eve可以用序列 0xEB 0xFE 替换两个指令字节，这在x86-64机器码中是一个两字节的指令，表示无条件的相对跳转，跳转距离为负两个字节。换句话说，这个指令将永远跳回自身，形成一个无限循环。此时 Eve 就可以垄断整个处理器，如何解决？

    • 解决方案：受限权限的内存映射

    幸运的是，我们计算机的硬件提供了一种机制，可以限制对内存不同区域的访问权限。这些权限是以每页来进行设置的：具体来说，每个内存页（回想一下，一个页是4 KB = 4096 字节的内存）可以设置多个访问位：

    • PTE_P: 当前。表示页面正在使用中。
    • PTE_W: 可写。表示页面可以读取和写入。
    • PTE_U: 用户空间。表示页面可以被用户空间进程访问（即，如果 %cs 设置为 3）。

    权限位存储在哪里？
    页面权限位存储在一个称为页面表的数据结构中。

    在 WeensyOS 内核中，内存权限在内核启动后的 kernel() 函数中设置。所有操作都在这个循环中完成：

    // (re-)initalize kernel page table:
// all of physical memory is accessible except `nullptr`
for (vmiter it(kernel_pagetable);
         it.va() < MEMSIZE_PHYSICAL;
         it += PAGESIZE) {
    if (it.va() != 0) {
        it.map(it.va(), PTE_P | PTE_W | PTE_U);
    } else {
        it.map(it.va(), 0);
    }
}
循环体做两件事情
1.如果我们不处理地址 0x0（空指针！），则在该地址上设置页面的所有权限。
2.如果我们处理空指针，清除所有权限，以便对页面的任何访问都会导致内核中的故障。（这对于捕获程序中的空指针解引用非常有帮助！）

    这个循环使用 C++ 迭代器语法来遍历存储在称为 “内核页表” 的东西中的内存映射。C++ 迭代器是一个方便的概念，用于遍历不支持简单整数索引访问的数据结构。该循环表示，对于一个新的内存迭代器 it ，类型为 vmiter ，在 kernel_pagetable 上，我们希望迭代直到当前地址（ it.va() ）为 MEMSIZE_PHYSICAL （在 WeensyOS 中为 0x20‘0000），并且我们希望以 PAGESIZE 的步长（4096 字节）跳转。（注意， vmiter 隐式地从地址 0 开始迭代； vmiter 类的构造函数的另一种语法是 vmiter it(kernel_pagetable, 0) 。）

    但是这里的内核内存权限太开放了！特别是在内核内存页面上设置 PTE_U 是危险的。
让我们将这个循环改为使内核内存对用户空间进程不可用，同时保持用户空间内存在 PROC_START_ADDR 以上对它们可用：
// (re-)initalize kernel page table:
// all of physical memory is accessible except `nullptr`
    for (vmiter it(kernel_pagetable);
         it.va() < MEMSIZE_PHYSICAL;
         it += PAGESIZE) {
        if (it.va() >= PROC_START_ADDR || it.va() == CONSOLE_ADDR) {
            // process memory, console (for memory-mapped I/O) accessible to userspace
            it.map(it.va(), PTE_P | PTE_W | PTE_U);
        } else if (it.va() != 0) {
            // kernel memory, NOT accessible to userspace
            it.map(it.va(), PTE_P | PTE_W);
        } else {
            // null page (for null pointer derefs) mapped with no permissions
            it.map(it.va(), 0);
        }
    }

    随着对内核的这个更改，Eve再也无法访问内核内存，我们达到了保护内存的目的。
